# 2026-06-16 ‚ÄĒ Project scaffolding

## What was done

Set up the **Explainers** repo: a no-build, static collection of interactive HTML
explainers (one self-contained `index.html` per topic), to be deployed on GitHub
Pages. Modeled on explainers.blog (Erik Kennedy) and Paras Chopra's explainers.

### Research
- Read both reference sites. Key findings: Paras' repo is **100% HTML** (CSS/JS
  inlined), each explainer a self-contained `index.html` in its own slug folder,
  with a root landing page of cards. Content philosophy (Erik Kennedy): "accurate
  AND accessible", "you don't understand it until you can predict it", concrete
  hook ‚Üí numbered question-headings ‚Üí intuition-before-math ‚Üí "well ackshually ūüßź"
  asides ‚Üí prediction section ‚Üí closure.

### Decisions (confirmed with user)
- Style guide lives in **`CLAUDE.md`** (intentionally overrides the global
  "keep CLAUDE.md empty" rule for this project).
- Explainers are **self-contained single files** (inline CSS+JS), matching Paras.
- Built **structure only** for now ‚ÄĒ no sample topic yet.

### Files created
- `CLAUDE.md` ‚ÄĒ the explainer-generation style guide (main deliverable).
- `template/index.html` ‚ÄĒ canonical skeleton: design tokens, reading layout, TOC,
  `setupCanvas()` demo harness (HiDPI + reduced-motion safe), example demo,
  "ackshually" `<details>`, prediction box, footer, commented KaTeX include.
- `index.html` ‚ÄĒ landing page rendering cards from an inline `EXPLAINERS`
  manifest; empty-state shown until the first explainer exists.
- `README.md`, `ARCHITECTURE.md`, `LICENSE` (MIT/digster), `.gitignore`,
  `.nojekyll`, `PROMPT.md`.
- `package.json`, `playwright.config.js`, `tests/explainers.spec.js` ‚ÄĒ Playwright
  smoke tests that auto-discover every explainer folder.

## How to continue
- To add an explainer: copy `template/` ‚Üí `<slug>/`, fill it in per `CLAUDE.md`,
  prepend a manifest entry in `index.html`, run `npm test`.
- Deploy: GitHub Pages ‚Üí Deploy from branch ‚Üí `main` / root.

## Update ‚ÄĒ landing-page card redesign + dark mode removed (same day)

Reworked the landing page article list and dropped dark mode site-wide (both
decided interactively via visual mockups before planning).

- `index.html`: replaced the `auto-fill` card **grid** (which stranded a lone card
  in empty tracks) with a single-column flex list capped at `max-width: 680px`.
  Cards are now full-width with a **corner-glow radial gradient**
  (`radial-gradient(130% 130% at 0% 0%, var(--t1), var(--bg) 62%)`), a 1px hue
  border, and the date as a small **hue-colored eyebrow above the title**. Added a
  six-class **hue palette** (`.hue-blue/teal/amber/violet/rose/green`, each sets
  `--c`/`--t1`/`--bd`) and a `PALETTE` array; the render loop assigns
  `PALETTE[i % PALETTE.length]` per card so color is auto-cycled (no per-entry
  work). Markup reordered so `.date` renders first.
- **Dark mode removed everywhere**: deleted the `@media (prefers-color-scheme:
  dark)` blocks from `index.html`, `template/index.html`, and `fractals/index.html`
  (site is now light-only). Reworded the fractals `readVar` comment (no longer "in
  dark mode"). Updated `CLAUDE.md`: the dark-mode technical-rule bullet became a
  "theme through the CSS variables" bullet noting the site is light-only, and the
  design-tokens reference no longer mentions a dark remap.
- Verified: `npm test` (3 pass, 0 console errors); browser checks at desktop +
  360px mobile, hue cycling across 6 injected cards, single-card no longer lonely.
  Screenshots removed after testing.
